Concept 105 / 22

Subagent / 并行小队

定义：主 Agent 派出的专职子任务执行者

目标不是“多开几个窗口”，而是清晰职责 + 并行执行 + 可汇总。

Subagent 的价值
  • 把大问题拆成可并行子问题，缩短总时长
  • 每个子任务保持单一维度，降低混乱上下文
  • 主 Agent 只负责策略与汇总，质量更稳定
典型拆分维度
  • 按目标：调研 / 设计 / 实现 / 验证
  • 按质量维度：correctness / performance / security
  • 按数据源：代码 / 文档 / 工单 / 监控
关键词：Parallel + Specialized + SummarizedSubagent
Subagent 实例06 / 22

Example

PR 评审：三个子队并行，主 Agent 汇总决策

A 队：Correctness
  • 检查边界条件与空值分支
  • 确认业务规则是否被破坏
B 队：Performance
  • 识别 N+1 查询和重复计算
  • 评估热点路径开销
C 队：Security
  • 检查权限边界和敏感信息泄露
  • 确认输入校验与输出过滤
主 Agent 输出 = 统一风险清单 + 优先级 + 修复顺序
这一步是“决策整合”，不是重复分析。
并行不是目的，可决策结果才是PR review pattern
Subagent 边界07 / 22

When to split / when not

拆分并不是越多越好：看独立性和耦合度

适合拆分
  • 子任务之间弱依赖，可独立完成
  • 需要多维度独立审视（如安全+性能）
  • 总任务复杂，单上下文容易失真
不适合拆分
  • 任务极小：几行修复即可完成
  • 步骤强依赖：后一阶段必须等待前一阶段
  • 成本大于收益：协调和汇总成本更高
检查点 1：职责是否单一？ 检查点 2：结果是否可独立验证？ 检查点 3：主 Agent 是否能无损汇总？
拆分原则：独立可验证 + 汇总可决策Subagent checklist
Concept 208 / 22

Hooks / 自动闸门

定义：在关键时机自动执行的约束与检查

Hooks 解决“知道该做但总会忘”的问题，把经验变成强制流程。

常见触发时机
  • 任务开始前：检查上下文和权限约束
  • 执行中：对高风险操作进行二次确认
  • 任务结束后：自动跑质量检查并记录摘要
典型收益
  • 减少漏测、漏扫、漏审这类低级失误
  • 统一团队行为，降低“个人习惯差异”
  • 把质量前移，在发布前就拦住问题
Hooks = 自动化守门员Process control
Hooks 实战09 / 22

Hook pipeline

把 Hook 放进“前中后”三段，质量成本最低

Before execution
预执行检查
校验工作目录、确认是否触发危险命令、读取必要上下文文件。
During execution
过程拦截
命中敏感模式（密钥、破坏性命令）时阻断并要求确认。
After execution
自动收尾
运行测试、产出摘要、提醒未完成校验项。
Hook 设计原则： 1) 只做可自动化、可确定的检查 2) 失败时给出可执行修复建议 3) 不把 Hook 变成“噪音提示器”
好 Hook：少而硬；坏 Hook：多而吵Hook patterns
Concept 310 / 22

MCP / Model Context Protocol

定义：让模型访问外部工具与实时数据的标准接口

模型层

负责推理、规划和自然语言交互。

协议层

把“调用能力”规范成稳定接口。

工具层

代码图谱、文档库、工单、内部 API。

为什么 MCP 是系统核心
  • 没有 MCP：回答可能“听起来对”，但难验证
  • 有了 MCP：结论可追溯到真实数据来源
  • 统一协议后：新工具接入成本可控，流程可扩展
MCP 让 Agent 从“会说”升级为“会查会做”MCP
MCP 实例11 / 22

Code intelligence example

同一个问题：有无 MCP，信息质量差一个数量级

没有 MCP
  • 靠记忆或盲搜拼接答案
  • 调用链容易漏节点或误判
  • 结论难复现、难审计
使用 MCP + CodeGraph
  • 先抓结构上下文，再拉关键源代码
  • 可直接追踪 from → to 的调用路径
  • 结果有出处，团队可复核
示例提问： “这个函数最终如何到达渲染层？” → 通过 trace/context/explore 组合，得到可验证路径而非猜测。
可验证性是工程化的分水岭MCP in action
Concept 412 / 22

Skill / 流程封装

定义：把高频任务沉淀成可触发的标准化命令

Skill 的本质
  • 不是快捷键，而是“带约束的执行模板”
  • 固化输入、输出、步骤和质量闸门
  • 让团队经验从“口口相传”变成“可复用资产”
典型触发方式
/article-writing 文章成稿流程 /code-review 按维度审查改动 /verify 验证改动是否真正生效
输入规范化 输出可预期 可持续迭代优化
Skill = 团队最佳实践的产品化Skill
Skill 设计13 / 22

How to design a good skill

一个可持续 Skill 至少包含四个部件

触发语义

什么时候必须触发，避免歧义。

输入契约

参数最小完备，缺失时提问补齐。

输出模板

结构固定，可直接交付或二次加工。

质量闸门

事实检查、格式校验、异常回退。

判断 Skill 是否成熟的信号
  • 新成员不依赖作者本人也能跑通
  • 多次执行结果结构稳定，差异可解释
  • 迭代成本低：改一处可全链路复用
成熟 Skill 的目标是“替代重复沟通”Skill quality
Concept 514 / 22

CLAUDE.md / 项目协作契约

定义：项目级长期指令，约束行为并统一输出风格

它不是“灵感提示词”，而是团队工作方式的书面协议。

CLAUDE.md 应该写什么
  • 沟通风格：先方案后解释、何时澄清
  • 风险边界：禁止密钥、危险命令确认机制
  • 检索策略：默认范围、何时使用 Git
  • 项目偏好：命名、注释、交付格式
它解决的核心问题
  • 不同会话中的行为一致性
  • 跨成员协作时的默认规范对齐
  • 减少“每次都重新交代规则”
规则写进文件，才能长期稳定执行CLAUDE.md
CLAUDE.md 示例15 / 22

Minimal but effective

一份能落地的 CLAUDE.md，通常像这样

## 沟通与执行 - 先给方案，再解释原因 - 需求不清时先提澄清问题 - 不输出半成品 ## 风险边界 - 禁止提交密钥 - 未确认不执行破坏性命令 ## 检索策略 - 默认做定向检索 - 未给路径先问范围
常见误区
  • 写成“愿景口号”，缺少可执行约束
  • 规则过多但无优先级，冲突时无法判定
  • 不维护：项目变化后文档长期过期
改进建议
  • 每条规则都可被验证、可被触发
  • 把高频反馈回写到文档中
好文档让行为一致；坏文档制造幻觉一致Contract quality

落地路线20 / 22

Adoption roadmap

从个人可用到团队可复制，建议分三层推进

L1：个人试用（1 周）
  • 先写最小 CLAUDE.md
  • 挑 1 个 Skill 反复打磨
  • 建立最基本 Hook 守门
L2：团队标准化（2-4 周）
  • 沉淀共用 Skill 库
  • 按任务类型定义 Subagent 模板
  • 统一 MCP 接入规范
L3：平台化（持续）
  • 把高价值 Hook 产品化
  • 追踪质量指标与返工率
  • 形成可审计交付流水线
先有规则，再谈效率 先做最小闭环，再扩规模 让每次反馈都可回写系统
路线图目标：降低随机性，提升可复制性Adoption

总结22 / 22

Takeaway

记住这五句，基本不会走偏

Subagent 做分工，Hooks 守质量，MCP 给事实，Skill 管复用，CLAUDE.md 保一致。
当这五件事同时在线，AI 才从“灵感助手”升级为“稳定产线”。
你可以马上做的三步
  • 今天：写最小版 CLAUDE.md
  • 本周：固化 1 个高频 Skill
  • 下周：给关键流程加 2 个 Hooks
扩展阅读方向
  • 多 Agent 编排策略（并行 vs 流水线）
  • MCP 能力目录与权限治理
  • 质量指标体系：缺陷率、返工率、交付时延
