# 把 Vibe Coding 做成“可交付系统”：Subagent、Hooks、MCP、Skill、CLAUDE.md 一次讲透

很多人把 vibe coding 理解成“想到哪写到哪”。

这会很快碰到三个问题：
- 结果不稳定（今天能跑，明天翻车）
- 上下文不一致（同一个需求，不同回合给出不同方案）
- 无法规模化（你一个人能用，团队接不住）

更实用的理解是：**vibe coding 不是放飞，而是把“自然语言开发”做成有边界、有分工、有质检的系统**。

下面按顺序讲清五个核心概念：
1. subagent
2. hooks
3. mcp
4. skill
5. claude.md

然后我们再把它们串成一条完整工作流。

---

## 1) Subagent：把一个大问题拆成并行小队

**定义**：Subagent 是主 Agent 派出去执行子任务的“专职助手”。

**作用**：
- 并行处理（加速）
- 角色分工（更专注）
- 降低主上下文负担（更稳）

**小例子**（PR 评审场景）：
- Subagent A：只看 correctness（逻辑是否错）
- Subagent B：只看 performance（是否引入明显性能损耗）
- Subagent C：只看 security（是否泄露敏感信息、权限边界）

主 Agent 最后只做一件事：汇总三路结论，给出优先级和修复建议。

> 关键点：Subagent 不是“多开几个窗口”，而是“明确职责 + 可并行执行”。

---

## 2) Hooks：给流程装自动闸门

**定义**：Hooks 是在特定时机自动触发的规则/脚本，用来约束 Agent 行为。

**作用**：
- 自动化守门（减少人为遗漏）
- 把团队规范变成可执行规则
- 在错误扩散前拦截

**小例子**：
- 在执行发布相关命令前，自动跑一遍测试
- 检测到敏感文件或密钥模式，直接阻断并报警
- 在任务结束时自动生成变更摘要

> 关键点：Hooks 解决的是“知道该做但总会忘”的问题。

---

## 3) MCP：给模型接上外部工具和真实世界数据

**定义**：MCP（Model Context Protocol）是模型访问外部能力的标准接口。

**作用**：
- 让 Agent 不只会“说”，还能“查”和“做”
- 把信息源从“训练记忆”升级为“实时可验证数据”
- 统一接入不同系统（文档库、代码索引、工单系统、内部 API）

**小例子**：
- 你问“这个函数谁在调用？”
- Agent 通过代码索引类 MCP 工具直接给出调用链
- 不再靠猜，不再靠全仓盲搜

> 关键点：没有 MCP，很多回答只是“像对的”；有 MCP，才能接近“可验证的对”。

---

## 4) Skill：把高频动作封装成可复用命令

**定义**：Skill 是一段可复用的任务模板/流程封装，通常通过 slash command 触发。

**作用**：
- 固化最佳实践
- 降低重复沟通成本
- 把“会的人”经验沉淀为“团队资产”

**小例子**：
- `/article-writing`：输入主题后，自动按既定结构产出可发布文章
- `/code-review`：按固定维度检查当前变更
- `/verify`：按预设步骤验证改动是否真的生效

> 关键点：Skill 是“流程产品化”。一次打磨，多次复用。

---

## 5) CLAUDE.md：项目级“操作说明书”

**定义**：CLAUDE.md 是项目内给 Agent 的长期指令文件，约定风格、边界和默认策略。

**作用**：
- 保持输出一致性
- 显式约束风险行为
- 减少每次都要重复说的规则

**小例子**：
```md
- 不允许提交任何密钥
- 不新增依赖，除非说明现有工具为何不够
- 先给方案，再解释原因
- 用户未给路径时，先问清范围再检索
```

> 关键点：CLAUDE.md 不是“提示词备忘录”，而是“团队协作契约”。

---

## 这五个概念如何配合成一套工作系统？

看一个完整场景：

**需求**：两天内交付一篇“支付系统重构复盘”技术文章，内容要准确、可发布、可复用。

### 第一步：用 CLAUDE.md 先定边界
先约束行为：禁止敏感信息、统一文风、明确检索策略、定义发布标准。

结果：后续所有动作都有统一“地基”。

### 第二步：用 Skill 启动标准流程
触发写作 Skill（例如 `/article-writing`），自动进入固定结构：提纲 → 证据收集 → 成稿 → 自检。

结果：不用每次从 0 设计流程。

### 第三步：用 Subagent 并行干活
主 Agent 拆 3 路：
- A：从代码和提交中提取技术事实
- B：从监控/工单整理故障时间线
- C：生成多个结构方案并比较可读性

结果：速度和质量一起提升。

### 第四步：用 MCP 拉取“可验证事实”
每路 Subagent 都通过 MCP 访问真实来源（代码索引、文档系统、工单数据）。

结果：文章里的关键论断有据可查，不靠印象写作。

### 第五步：用 Hooks 做自动质检
在出稿或发布前自动执行：
- 敏感信息扫描
- 链接可用性检查
- 格式规范检查

结果：把“低级失误”挡在发布前。

### 最后：主 Agent 汇总并交付
主 Agent 只做高价值动作：
- 合并多路结果
- 去冲突
- 输出最终可发布版本

结果：你得到的是“成稿”，不是“半成品草稿”。

---

## 一句话工作流

```text
CLAUDE.md 定规则
→ Skill 拉起标准流程
→ Subagent 并行执行
→ MCP 提供真实数据与外部能力
→ Hooks 自动质检兜底
→ 主 Agent 汇总交付
```

这就是可复制的 vibe coding：**前端是自然语言，后端是工程系统**。

---

## 结尾总结

如果你只记住一件事：

**vibe coding 的核心不是“更会聊天”，而是“把聊天背后变成可控流水线”。**

- Subagent 解决分工与并行
- Hooks 解决流程守门
- MCP 解决事实与能力接入
- Skill 解决复用与规模化
- CLAUDE.md 解决长期一致性

当这五件事同时在线，AI 才从“灵感助手”变成“稳定产线”。
